# 本地版本管理规则片段

这是可合并进项目 agent 规则文件的片段，不是完整替换文件。根据当前 agent 的习惯，可以合并进 `AGENTS.md` 或 `CLAUDE.md`。如果规则文件已经存在，把本节合并进去，不要覆盖原有规则。

## 修改已有文件前后

- 只在修改本地已有文件时使用本规则；只创建全新文件且不覆盖已有文件时，不需要记录版本。
- 在修改本地已有文件前，先记录一次当前版本。
- 修改完成后，再记录一次改完后的版本。
- 如果项目已安装 `$version-management` skill，优先使用它完成这两次记录。
- 如果没有安装该 skill，也必须执行同等保护：检查当前文件夹是否已经有本地版本记录，并在修改前和修改后分别记录版本。
- 如果当前文件夹还没有本地版本记录，先询问用户：  
  “这个文件夹还没有版本记录。我可以先记录当前状态，这样改错了可以回到现在这个版本。要我现在设置吗？”
- 只有用户同意后，才能首次设置本地版本记录；设置后，先记录当前状态，再开始修改文件。
- 如果电脑没有安装 git，先询问用户：  
  “你的电脑还没有安装用于版本管理的基础工具。要我帮你安装吗？安装后我就可以在改文件前后记录版本，方便之后回退。”

## 多轮对话和其他 skill

- 很多轮对话后，只要再次修改本地已有文件，仍然要在本轮修改前后各记录一次。
- 如果同时使用文档、PPT、图片、代码等其他 skill，本规则作为外层保护，不替代其他 skill 的编辑流程。
- 恢复、还原、回退已有文件时，也按本规则执行：先说明影响范围，获得确认，再恢复。

## 和用户沟通

- 面向普通用户时，说“记录当前版本”“查看这次改了什么”“回到之前版本”。
- 除非用户主动询问技术细节，否则不要说 commit、branch、staging、HEAD、stash 等 git 术语。
- 完成修改后，简要说明改了哪些文件，并告诉用户可以要求回到 AI 修改前或修改后的版本。

## 安全边界

- 不要丢弃、覆盖、重置或清理用户已有改动，除非用户明确要求。
- 不要推送、同步、发布或修改远程仓库设置。
- 回退文件前，先说明会回到哪个版本、影响哪些文件，并获得明确确认。
- 如果需要使用底层命令完成记录或回退，只在 agent 内部使用；面向用户仍然使用“版本记录”和“回退”的说法。
